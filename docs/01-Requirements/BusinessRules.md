## 1. 角色权限边界

### 1.1 角色定义

本系统定义三类核心角色，各角色具有明确的职责边界和权限范围：

| 角色 | 角色标识 | 职责定位 | 数据可见范围 |
|------|----------|----------|--------------|
| 消费者 | `CONSUMER` | 线上购物主体 | 仅可见自身订单、地址、个人信息 |
| 门店管理员 | `STORE_ADMIN` | 单店运营管理者 | 仅可见所属门店的商品、库存、订单 |
| 系统管理员 | `SYSTEM_ADMIN` | 平台全局管理者 | 可见全平台所有数据 |

### 1.2 消费者权限边界

**允许的操作：**
- 浏览平台商品目录及门店商品详情
- 查看门店信息（营业状态、地址、营业时间）
- 将商品加入购物车
- 选择履约门店并提交订单
- 查看、取消自己的订单
- 管理个人收货地址
- 修改个人基本信息

**禁止的操作：**
- 访问其他消费者的任何信息
- 修改商品信息、价格、库存
- 访问门店管理后台
- 查看平台运营数据

**边界约束：**
- 消费者下单时必须选择一个有效的履约门店
- 消费者只能取消处于"待支付"或"待接单"状态的订单
- 消费者地址数量上限为 20 条

### 1.3 门店管理员权限边界

**允许的操作：**
- 管理本门店商品的库存数量
- 设置本门店商品的销售价格（可与平台建议价不同）
- 上架/下架本门店的商品
- 接单、拒单、标记出货
- 核销自提订单
- 查看本门店的销售统计数据

**禁止的操作：**
- 访问其他门店的任何数据
- 创建或删除平台商品（只能从平台商品库选择上架）
- 修改平台级商品基础信息（名称、图片、描述）
- 管理其他门店的订单
- 访问系统管理后台
- 查看全平台统计数据

**边界约束：**
- 一个门店管理员账号只能绑定一个门店
- 门店管理员设置的价格不得低于平台最低限价（如有）
- 门店管理员必须在规定时间内处理订单（超时系统自动取消）

### 1.4 系统管理员权限边界

**允许的操作：**
- 管理平台商品库（创建、编辑、删除商品）
- 管理所有门店信息（创建、编辑、启用/禁用门店）
- 管理所有用户账号（创建、编辑、禁用账号）
- 分配门店管理员角色
- 查看全平台销售数据统计
- 配置系统参数（订单超时时间、库存预警阈值等）

**禁止的操作：**
- 代替消费者下单
- 直接修改门店库存（需通过门店管理员操作）
- 删除已产生交易的历史数据

**边界约束：**
- 系统管理员操作需记录操作日志
- 敏感操作（删除门店、禁用账号）需二次确认

### 1.5 权限矩阵汇总

| 功能模块 | 消费者 | 门店管理员 | 系统管理员 |
|----------|--------|------------|------------|

## 2. 商品与门店库存价格关系

### 2.1 双层商品架构

本系统采用**平台商品 + 门店商品**的双层架构设计：

```
┌─────────────────────────────────────────────────────────┐
│                    平台商品库 (SPU)                      │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐    │
│  │ 商品A   │  │ 商品B   │  │ 商品C   │  │ 商品D   │    │
│  │ 乐事薯片│  │ 奥利奥  │  │ 旺仔牛奶│  │ 德芙巧克力│   │
│  └────┬────┘  └────┬────┘  └────┬────┘  └────┬────┘    │
└───────┼────────────┼────────────┼────────────┼─────────┘
        │            │            │            │
        ▼            ▼            ▼            ▼
┌─────────────────────────────────────────────────────────┐
│                   门店商品层 (SKU)                       │
│  ┌──────────────────────────────────────────────────┐   │
│  │ 门店1（中关村店）                                  │   │
│  │  商品A: 库存50, 价格8.5元                         │   │
│  │  商品B: 库存30, 价格12.0元                        │   │
│  │  商品C: 未上架                                    │   │
│  └──────────────────────────────────────────────────┘   │
│  ┌──────────────────────────────────────────────────┐   │
│  │ 门店2（望京店）                                    │   │
│  │  商品A: 库存20, 价格9.0元                         │   │
│  │  商品B: 库存0, 价格12.0元（售罄）                  │   │
│  │  商品D: 库存100, 价格25.0元                       │   │
│  └──────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

### 2.2 平台商品（SPU）

**定义**：平台商品是系统管理员维护的商品基础信息，代表一个标准化的商品定义。

**包含属性**：
- 商品基础信息：名称、描述、图片、分类
- 平台建议零售价（参考价）
- 商品状态：启用/禁用
- 商品规格信息

**业务规则**：
- 平台商品由系统管理员统一创建和维护
- 门店只能从平台商品库中选择商品进行上架
- 平台商品禁用后，所有门店该商品自动下架
- 平台商品不直接参与交易，消费者购买的是门店商品

### 2.3 门店商品（SKU）

**定义**：门店商品是平台商品在具体门店的实例化，包含该门店特有的库存和价格信息。

**包含属性**：
- 关联的平台商品ID
- 关联的门店ID
- 门店销售价格
- 门店库存数量
- 上架状态：上架/下架

**业务规则**：
- 门店商品的价格由门店管理员自主设定
- 不同门店同一商品可以有不同的价格
- 门店库存独立管理，各门店库存互不影响
- 门店商品下架后，消费者在该门店无法购买该商品
- 库存为0时，商品显示"售罄"但不自动下架

### 2.4 价格规则

| 规则项 | 说明 |
|--------|------|
| 价格设定权 | 门店管理员拥有本店商品定价权 |
| 价格下限 | 门店价格不得低于平台设定的最低限价（如有） |
| 价格展示 | 消费者看到的是所选门店的实际销售价格 |
| 价格变更 | 价格变更即时生效，不影响已下单订单 |
| 购物车价格 | 购物车中商品价格随门店价格实时更新 |

### 2.5 库存规则

| 规则项 | 说明 |
|--------|------|
| 库存归属 | 库存归属于具体门店，非平台统一库存 |
| 库存扣减 | 消费者下单成功后扣减对应门店库存 |
| 库存回滚 | 订单取消后库存自动回滚 |
| 库存预警 | 库存低于阈值时向门店管理员发送提醒 |
| 超卖防护 | 下单时校验库存，库存不足则下单失败 |

## 3. 下单绑定履约门店与自提核销流程

### 3.1 履约门店绑定规则

**门店选择时机**：消费者在下单前必须选择一个履约门店。

**门店选择方式**：
1. **基于定位推荐**：系统根据消费者位置推荐最近的门店
2. **手动选择**：消费者可从门店列表中自主选择
3. **购物车绑定**：购物车与门店绑定，切换门店需清空购物车

**绑定约束**：
- 一个订单只能绑定一个履约门店
- 订单提交后履约门店不可变更
- 门店必须处于营业状态才可被选择
- 所选商品必须在该门店有库存且已上架

### 3.3 自提核销流程

- 订单支付成功后生成唯一自提码
- 自提码格式：8位数字 + 字母组合
- 自提码有效期：订单完成前持续有效
- 自提码展示：订单详情页 + 短信通知

**核销业务规则**：
| 规则项 | 说明 |
|--------|------|
| 核销权限 | 仅订单所属门店的管理员可核销 |
| 核销时机 | 订单状态为"待自提"时可核销 |
| 核销方式 | 扫描二维码或手动输入自提码 |
| 核销结果 | 核销成功后订单状态变为"已完成" |
| 重复核销 | 系统阻止重复核销并提示 |
| 核销记录 | 记录核销时间、操作人 |

## 4. 订单状态机

### 4.1 订单状态定义

| 状态码 | 状态名称 | 状态说明 |
|--------|----------|----------|
| `PENDING_PAYMENT` | 待支付 | 订单已创建，等待消费者支付 |
| `PENDING_ACCEPT` | 待接单 | 支付成功，等待门店接单 |
| `ACCEPTED` | 已接单 | 门店已接单，准备商品中 |
| `READY_FOR_PICKUP` | 待自提 | 商品已备好，等待消费者到店自提 |
| `COMPLETED` | 已完成 | 消费者已取货，订单完成 |
| `CANCELLED` | 已取消 | 订单已取消 |
| `REFUNDED` | 已退款 | 订单已退款 |

### 4.3 状态流转事件

| 当前状态 | 事件 | 目标状态 | 触发者 | 附加动作 |
|----------|------|----------|--------|----------|
| 待支付 | 支付成功 | 待接单 | 系统（支付回调） | 生成自提码 |
| 待支付 | 用户取消 | 已取消 | 消费者 | 库存回滚 |
| 待支付 | 支付超时 | 已取消 | 系统（定时任务） | 库存回滚 |
| 待接单 | 门店接单 | 已接单 | 门店管理员 | - |
| 待接单 | 门店拒单 | 已取消 | 门店管理员 | 触发退款、库存回滚 |
| 待接单 | 接单超时 | 已取消 | 系统（定时任务） | 触发退款、库存回滚 |
| 待接单 | 用户取消 | 已取消 | 消费者 | 触发退款、库存回滚 |
| 已接单 | 备货完成 | 待自提 | 门店管理员 | 通知消费者 |
| 待自提 | 核销成功 | 已完成 | 门店管理员 | 记录核销信息 |
| 待自提 | 自提超时 | 已取消 | 系统（定时任务） | 触发退款、库存回滚 |

### 4.4 异常分支处理

#### 4.4.1 取消场景

| 取消场景 | 触发条件 | 处理逻辑 |
|----------|----------|----------|
| 用户主动取消（未支付） | 订单状态=待支付 | 直接取消，库存回滚 |
| 用户主动取消（已支付） | 订单状态=待接单 | 取消订单，自动退款，库存回滚 |
| 门店拒单 | 订单状态=待接单 | 取消订单，自动退款，库存回滚，记录拒单原因 |
| 用户取消（已接单后） | 订单状态=已接单/待自提 | 不允许取消，需联系客服 |

#### 4.4.2 超时场景

| 超时场景 | 超时时长 | 处理逻辑 |
|----------|----------|----------|
| 支付超时 | 30分钟 | 自动取消订单，库存回滚 |
| 接单超时 | 15分钟 | 自动取消订单，自动退款，库存回滚 |
| 自提超时 | 24小时 | 自动取消订单，自动退款，库存回滚 |

#### 4.4.3 状态流转约束

- 状态只能按照定义的路径流转，不允许跳跃
- 已完成和已取消是终态，不可再变更
- 每次状态变更需记录变更时间和操作人

## 5. 库存一致性策略

### 5.1 MVP阶段策略：下单扣库存

**策略说明**：消费者提交订单时立即扣减库存，订单取消时回滚库存。

**流程图**：

```
┌─────────────┐
│  提交订单   │
└──────┬──────┘
       │
       ▼
┌─────────────┐     ┌─────────────┐
│  查询库存   │────▶│ 库存 >= 购买数量？│
└─────────────┘     └──────┬──────┘
                           │
         ┌─────────────────┼─────────────────┐
         ▼                                   ▼
        是                                  否
         │                                   │
         ▼                                   ▼
  ┌─────────────┐                     ┌─────────────┐
  │  扣减库存   │                     │  下单失败   │
  │ stock -= qty│                     │  提示库存   │
  └──────┬──────┘                     │  不足       │
         │                            └─────────────┘
         ▼
  ┌─────────────┐
  │  创建订单   │
  └──────┬──────┘
         │
         ▼
  ┌─────────────┐
  │  订单取消？ │
  └──────┬──────┘
         │
    ┌────┴────┐
    ▼         ▼
   是        否
    │         │
    ▼         ▼
┌─────────┐  ┌─────────┐
│回滚库存 │  │ 正常流程│
│stock+=qty│ └─────────┘
└─────────┘
```

**优点**：
- 实现简单，逻辑清晰
- 不会出现超卖问题

**缺点**：
- 可能出现恶意占用库存（下单不支付）
- 支付超时前库存被锁定

**适用场景**：MVP阶段，订单量较小，用户行为较规范

### 5.2 进阶策略：库存锁定机制

**策略说明**：下单时锁定库存（预扣），支付成功后正式扣减，超时或取消则释放锁定。

**库存字段设计**：
```
total_stock     -- 总库存
locked_stock    -- 锁定库存（已下单未支付）
available_stock -- 可用库存 = total_stock - locked_stock
```

**流程图**：

```
┌─────────────┐
│  提交订单   │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 可用库存>=购买数量？│
└──────┬──────┘
       │
  ┌────┴────┐
  ▼         ▼
 是        否
  │         │
  ▼         ▼
┌─────────┐ ┌─────────┐
│锁定库存 │ │下单失败 │
│locked+=qty│└─────────┘
└────┬────┘
     │
     ▼
┌─────────────┐
│  等待支付   │
└──────┬──────┘
       │
  ┌────┴────────────┐
  ▼                 ▼
支付成功          支付超时/取消
  │                 │
  ▼                 ▼
┌─────────┐     ┌─────────┐
│正式扣减 │     │释放锁定 │
│total-=qty│    │locked-=qty│
│locked-=qty│   └─────────┘
└─────────┘
```

**优点**：
- 防止恶意占用库存
- 库存利用率更高
- 支持更复杂的业务场景

**缺点**：
- 实现复杂度增加
- 需要定时任务处理超时释放

### 5.3 并发控制

**数据库层面**：
- 使用乐观锁（版本号）或悲观锁（SELECT FOR UPDATE）
- 库存扣减使用原子操作：`UPDATE SET stock = stock - ? WHERE stock >= ?`

**应用层面**：
- 高并发场景可引入Redis预扣库存
- 使用分布式锁防止重复扣减

### 5.4 库存回滚触发条件

| 触发场景 | 回滚时机 | 回滚方式 |
|----------|----------|----------|
| 支付超时 | 定时任务扫描 | 自动回滚 |
| 用户取消（未支付） | 用户操作时 | 即时回滚 |
| 用户取消（已支付） | 退款成功后 | 退款回调触发 |
| 门店拒单 | 拒单操作时 | 即时回滚 |
| 接单超时 | 定时任务扫描 | 自动回滚 |
| 自提超时 | 定时任务扫描 | 自动回滚 |
